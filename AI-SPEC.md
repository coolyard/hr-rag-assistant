# AI-SPEC.md — AI 能力规范与行为约束

> 本文档定义 AI 能做什么、不能做什么、以及怎么做。是防止幻觉、保证回答质量的**核心宪法**。

---

## 1. 知识边界（Knowledge Boundary）

### 1.1 知识源
- **唯一知识源**：`docs/hr-documents/` 目录下的 HR Markdown 文档（包括内置文档和用户上传的文档）
- **禁止引用**：外部互联网知识、通用常识、模型预训练知识

### 1.2 分类体系
| 分类标识 | 中文名 | 对应文档 | 颜色标识 |
|---------|--------|---------|---------|
| `annual_leave` | 年假 | `年假制度.md` | 浅蓝 `#E3F2FD` |
| `reimbursement` | 报销 | `报销流程.md` | 浅绿 `#E8F5E9` |
| `promotion` | 晋升 | `晋升规则.md` | 浅橙 `#FFF3E0` |
| `attendance` | 考勤 | `考勤制度.md` | 浅紫 `#F3E5F5` |
| `welfare` | 福利 | `员工福利.md` | 浅黄 `#FFFDE7` |
| `custom` | 自定义 | 用户上传的文档 | 灰色 `#F5F5F5` |

### 1.3 拒绝策略
当检索结果无法回答问题，或问题超出知识边界时，必须返回：

```
根据现有 HR 文档，无法确认该问题的答案。建议联系 HR 部门获取准确信息。
```

**触发拒绝的条件**（满足任一）：
1. 向量检索 Top-K 结果中，最高相似度 < 0.5
2. 关键词检索无匹配且向量检索最高相似度 < 0.5
3. 用户问题涉及**他人**隐私（如"张三的工资是多少"）——查询自己的个人数据（如"我有多少天年假"）不触发此条件，属于"个人数据查询"允许范围
4. 用户问题涉及公司机密（如"下季度裁员计划"）
5. 用户问题与 HR 制度无关（如"今天天气怎么样"）

> **个人数据查询 vs 他人隐私**：用户查询自己的年假、报销、考勤等个人数据是允许的（通过 UserProfile 注入机制回答）。查询其他员工的个人信息属于隐私侵犯，必须拒绝。

## 2. RAG Pipeline 参数规范

### 2.1 检索参数（不可随意修改）
| 参数 | 值 | 说明 |
|------|-----|------|
| `embedding_model` | `nomic-embed-text` | Ollama 本地模型，768 维 |
| `vector_top_k` | 3 | 向量检索返回片段数 |
| `keyword_top_k` | 3 | 关键词检索返回片段数 |
| `merge_top_k` | 3 | 合并去重后最终片段数 |
| `similarity_threshold` | 0.5 | 最低相似度阈值 |
| `vector_weight` | 0.4 | 混合检索中向量检索权重 |
| `keyword_weight` | 0.6 | 混合检索中关键词检索权重 |

### 2.2 混合检索逻辑
1. **向量检索**：将用户问题通过 `nomic-embed-text` 生成 768 维向量，与文档片段向量计算余弦相似度，取 Top-3
2. **关键词检索**：预定义 30+ 个中文 HR 关键词，从问题中提取匹配词，对文档片段统计词频 + 标题匹配加分，取 Top-3
3. **合并去重**：按加权分数（向量 0.4 + 关键词 0.6）排序，去重后取最终 Top-3

### 2.3 上下文窗口管理
- **多轮对话保留**：最近 5 轮对话历史
- **检索上下文**：合并后的 Top-3 文档片段
- **总上下文**：System Prompt + 5 轮历史 + 检索片段 + 当前问题
- **截断策略**：如果总 Token 数超过模型上下文限制，优先保留检索片段，压缩历史对话（保留最近 3 轮）

## 3. Prompt 工程规范

### 3.1 System Prompt 模板（固定不可变）

```
你是企业 HR 助手，专门回答员工关于公司制度、政策和流程的问题。

## 核心规则
1. 【知识边界】你只能基于以下检索到的 HR 文档片段回答问题，禁止引用外部知识或推测。
2. 【准确性优先】如果文档片段无法完整回答问题，或你不确定，必须明确告知"根据现有 HR 文档，无法确认该问题的答案"。
3. 【来源引用】每个回答必须标注来源文档名称（如"《年假制度》"）。
4. 【隐私保护】涉及**他人**隐私（如具体员工姓名、工资数字）的问题，拒绝回答并提示联系 HR。
5. 【个人数据】如果下方提供了"当前用户个人信息"，用户询问自己的数据时（如"我有多少天年假"），优先基于个人数据回答，并用文档片段补充制度依据。
6. 【语气】使用中文，语气专业、简洁、友好。

## 检索到的文档片段
{{retrieved_chunks}}

## 当前用户个人信息（仅在用户询问自己数据时提供）
{{user_profile}}

## 对话历史
{{conversation_history}}

## 当前问题
{{user_question}}

请基于以上文档片段回答问题。如果文档片段为空或无关，请直接返回拒绝话术。
```

### 3.2 拒绝话术模板（固定不可变）

```
根据现有 HR 文档，无法确认该问题的答案。建议联系 HR 部门获取准确信息。
```

## 4. 幻觉控制策略（四层防御）

### Layer 1: 检索阈值过滤（Pre-filter）
- 向量检索 + 关键词检索合并后，最高相似度 < 0.5 的查询，**不进入 LLM 生成阶段**
- 直接返回拒绝话术

### Layer 2: System Prompt 强制约束（In-prompt）
- System Prompt 中明确写入"只能基于检索片段回答"
- 要求 LLM 标注来源

### Layer 3: 回答后校验（Post-hoc）
- 简单规则校验：检查回答中是否包含文档片段外的具体数字/日期/人名
- 如果检测到疑似幻觉内容，降级为保守表达

### Layer 4: 置信度标签（Transparency）
- 🟢 **高置信度**：检索相似度 > 0.8
- 🟡 **中置信度**：检索相似度 0.5-0.8
- 🔴 **低置信度/拒绝**：检索相似度 < 0.5

## 5. LLM 调用规范

### 5.1 模型配置
| 参数 | 值 | 说明 |
|------|-----|------|
| `model` | `qwen2.5:7b-instruct` | Ollama 本地模型 |
| `temperature` | 0.3 | 低温度，保证事实性 |
| `top_p` | 0.9 | 核采样 |
| `max_tokens` | 1024 | 单次回答最大长度 |
| `stream` | `true` | 必须开启流式输出 |

### 5.2 流式输出协议
- 使用 SSE (Server-Sent Events) 推送
- 格式：`data: {"chunk": "...", "done": false}`
- 结束标记：`data: {"chunk": "", "done": true, "sources": [...]}`

## 6. Embedding 服务规范

### 6.1 模型配置
| 参数 | 值 |
|------|-----|
| `model` | `nomic-embed-text` |
| `dimensions` | 768 |
| `normalize` | `true` |

### 6.2 文档分块策略
- 按 Markdown 二级标题（`##`）分块
- 每个 chunk 最大 512 字符，重叠 50 字符
- 过滤长度 < 20 字符的片段
- 每个文档最多取前 10 个 chunk
- **上传新文档后，自动触发分块和 Embedding 索引重建**
